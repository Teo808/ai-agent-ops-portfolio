Case Study 02

I treat agent memory like reliability infrastructure.

My memory-system plan combines human-readable Obsidian notes with Supermemory-style recall so an agent can use context without mixing sources, projects, or stale facts.

Problem This Solves

Agent memory becomes risky when it is treated like a single bucket of facts. A useful memory system needs source labels, project boundaries, confidence levels, and currentness checks so the agent can avoid mixing stale personal context with active project work.

Design Principles

Source every memoryI want recall to be explainable and auditable.
Scope by projectI prevent broad personal history from overriding current work.
Track currentnessI mark old setup instructions as stale after tools change.
Save verified lessonsI reduce noise and prevent false confidence.
Recall rules
  • Prefer current project memories over broad personal memories.
  • Label whether something is confirmed, inferred, planned, or stale.
  • Keep implementation notes separate from preferences.
  • Do not use private facts unless they directly help the task.
Quality controls
  • Every memory should have a project, source, and date.
  • A setup memory should include the command or UI path that proves it.
  • A lesson should not be saved until the fix is checked.
  • Sensitive information should be summarized without exposing secrets.

Hiring Signal

Forward deployed agent work often depends on whether the agent can retrieve the right context for the right environment. This plan shows how I think about memory as part of operational reliability, not just a convenience feature.
